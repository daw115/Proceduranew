Przebieg nominalny (stan poprawny) — krok po kroku

Operacyjna esencja dyżuru wg Procedury skróconej v4.1 i QuickRef (HLBP CCCt 4.2 v1). Warunki wstępne: Initial FB Computation zakończony (potwierdzenie CCC), otwarty pulpit „IDCC FB Pulpit dyspozytorski" w CCM, dostęp do skrzynki kdm6@pse.pl, aktywny login w Perun4V.

Sześć kroków iteracji IDCC(b)/(c)/(d)

  1. CCM: agregaty PERUN_IDxx (FBA) i PERUN_IDxx_ATC (ATC) → zielone; kolumna sFTP/ZP wysł. → zielona.
  2. Perun4V (lccv.tscnet.eu): typ ID CCCt, sprawdź Business Day; user sftp@pse.pl. Timestampy kompletne: b=0/24/0/25, c=0/18/0/19, d=0/12/0/13 (0 Failed).
  3. CCM: IVA FIDxx-710 → CCtool wysł. + CCtool zwalid. zielone; Raporty IDxx → MinIO zielony.
  4. CCM: ATC Based Validation FIDxx-928 → CCtool wysł. zielony w terminie TET.
  5. kdm6@pse.pl: raport CCA → sprawdź niepoliczone TS i wartości IVA; oceń zasadność (FBA i ATC).
  6. [jeśli dotyczy] IVA BACKUP → PPM → „Wyślij do CCTool" → opis → Tak; CCtool zwalid. zielony po ~30 s.

IDCC(a) — monitoring FID1-928 (bez Perun4V)

W IDCC(a) nie ma paczki Perun ani IVA FBA — PSE monitoruje wyłącznie plik zwrotny FID1-928 w CCM. Wersja v1 (~10:00 D-1) zawiera wartości zerowe (fallback); po SDAC narzędzie CCA sprawdza CNEC-e PL i publikowana jest wersja v2 z wynikiem walidacji ATC.

  1. A.1. CCM: wiersz „ATC Based Validation (FID1-928)". Po 13:15 D-1 wersja v2 zielona (v1 = wartości zerowe).
  2. A.2. Brak v2 po 14:30 D-1 → sprawdź MinIO (cca/ID_FBCC/out/BD) → poinformuj telefonicznie CCC.

IVA BACKUP — zasady

  • v1 = backup na WSZYSTKIE TS; v2+ = wyniki Perun uzupełnione backupem na niepoliczone TS.
  • Sygnał na pulpicie: zielony kafelek z pulsującą czerwoną otoczką.
  • Wysyłka: PPM → „Wyślij do CCTool" → opis → Tak. Wersja BACKUP musi być wyższa niż IVA z Perun (blokada dla tej samej lub niższej).
  • ERR-I (część TS niepoliczona): czekaj na v2+ — NIE wysyłaj v1!
  • Process failed (0 TS): wyślij v1; jeśli BACKUP niedostępny (kafelek szary) — telefon do CCC natychmiast. W IDCC nie ma fallbacku 20% Fmax.

ERR-I — błędy na wybranych TS

  1. Perun4V: Status → Details → [Result] → zanotuj Failed TS (pamiętaj: 01:30 Perun = godz. 2 CCCt).
  2. Czekaj na IVA BACKUP v2+ (Wersja ≥ 2 w CCM). Nie wysyłaj v1.
  3. PPM na IVA BACKUP → „Wyślij do CCTool" → opis (niepoliczone TS) → Tak.
  4. CCtool zwalid. zielony po ~30 s → koniec.

Process failed — wszystkie TS niepoliczone

  1. Sprawdź e-mail kdm6 i CCM. Czy IVA BACKUP dostępna (zielona/fioletowa)?
  2. Tak → PPM → „Wyślij do CCTool" → opis: „Process failed – v1" → Tak. CCtool zwalid. zielony.
  3. Nie (kafelek szary) → poinformuj CCC natychmiast telefonicznie. Brak fallbacku 20% Fmax — brak dalszych działań PSE.

B1 — Brak transferu SFTP (>30 min od „sFTP wysł.")

  1. CCM: LPM na agregat PERUN_IDxx → „Pobierz ZIP". Maska: PSE_RRRRMMDD_IDx_v*_input_validation_IDx.zip.
  2. Perun4V: Upload ZIP → sprawdź BD i liczbę TS → „Estimate security reserve" → komunikat „Calculation started".
  3. Po zakończeniu: F710 V (lub Status → „Download output files") → plik FIDxx-710.
  4. CCM: LPM na IVA FIDxx-710 → wyślij; LPM na Raporty IDxx z Perun → wyślij.
  5. Powtórz cały cykl 1–4 dla paczki ATC (PERUN_IDxx_ATC).

B2 — Za mały plik (czerwony „!")

  1. LPM na kafelek z „!" → „Informacje" → zanotuj numer pliku.
  2. CCCt: Message Viewer → wpisz numer (665=CB, 620=CGM, 701=Vertices) → Apply → pobierz (Sender=PSE).
  3. CCM: LPM na składową → wyślij. Po CET: „wyślij po CET" (status ciemnozielony W).

B3 — Czarny agregat (brak paczki ZIP)

  1. Rozwiń agregat w CCM → szare kafelki = brakujące składowe.
  2. CCCt: Message Viewer → pobierz brakujące (numery: 610, 620, 632, 645, 657, 659, 665, 701).
  3. CCM: LPM → wyślij każdą składową. Gdy wszystkie zielone → Connector tworzy ZIP → „Pobierz" → dalej wg B1.

B4 — Ręczny upload IVA do CCCt (awaria ECP/EDX)

  1. MinIO: bucket perun/ID_FBCC/out/yyyy-mm-dd → plik FIDxx-710.
  2. CCCt: Message Viewer → „Manual Upload" → wskaż FIDxx-710 → „Upload file".
  3. Odśwież: Message Definition „310" → Apply. Sender=PSE → status Processed ✓.
