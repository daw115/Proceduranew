IDCC(d) — przebieg procesu

Rysunek 2 ilustruje sekwencję działań, które poszczególne strony zaangażowane w proces pre-couplingu IDCC(d) muszą wykonać. Ten ogólny przegląd pokazuje również jasno, jakiego rodzaju informacje każdy partner musi wysłać i otrzymać.

Rysunek 2: Wysokopoziomowy przepływ procesu dla IDCC(d) (tylko pre-coupling) z orientacyjnymi czasami.

Niniejszy rozdział opisuje skoordynowane przygotowanie ograniczeń sieciowych dla IDCC(d), które są dostarczane do CCCt. Działania tej fazy obejmują:

  • IDCF (Intraday Congestion Forecast): TSO dostarczają IGM-y do RCC (Coreso, TSCNET), które łączą IGM-y i tworzą wspólny model sieci (CGM). Modele CGM są ulepszane w zależności od RA podjętych podczas procesu CSA (Coordinated Security Analysis).
  • Initial FB Computation (Wstępne obliczenie FB): wykonywane w CCCt przez CCC, na podstawie danych wejściowych od TSO i RCC.
  • Individual validation (Walidacja indywidualna): krok wykonywany przez każdy TSO w celu walidacji wstępnych wyników FB (z możliwością ich korekty).
  • Final FB Computation (Końcowe obliczenie FB): obliczenie wykonywane w CCCt przez CCC, na podstawie zaktualizowanych plików CB oraz EC/ATC przesłanych przez Core TSO.

Poniższe podsekcje opisują proces normalny oraz wskazują powiązania z procesami backup i fallback dla IDCC(d). Procesy backup są szczegółowo opisane na końcu każdego rozdziału w szarej tabeli. Bardziej szczegółowe opisy znajdują się w odpowiednich podsekcjach IDCC(b) (rozdział 6).

Wstępne obliczenie flow-based (Initial flow-based computation)

CCCt oblicza wstępne parametry FB, tj. PTDF i RAM dla każdego wstępnego CNEC. Wstępna połączona lista CNEC jest tworzona przez połączenie indywidualnych list CNEC wszystkich Core TSO. Następnie wykonywana jest selekcja CNEC w celu usunięcia tych CNEC, dla których maksymalny PTDF strefa-do-strefy (zone-to-zone) dla wszystkich wymian w Core jest mniejszy niż 5%. CNEC z maksymalnym PTDF strefa-do-strefy równym lub większym niż 5% stanowią końcową listę CNEC. To sprawdzenie jest wykonywane, a spełnienie kryterium 5% jest sprawdzane oddzielnie dla każdego timestampu dnia handlowego (business day). Na końcu obliczenia wyniki są zapisywane przez CCCt i wysyłane do TSO. Dedykowane wyniki, takie jak zaktualizowany plik CB, są wysyłane do CCC.

Ekstrakcja ID ATC (ID ATC Extraction)

Ostatnim krokiem przed wysłaniem wyników obliczania zdolności do procesu coupling jest ekstrakcja ID ATC (oraz powiązanego NTC, ponieważ Coupling Mechanism używa NTC i AAC w swoim procesie coupling). ID ATC składa się z trzech kroków: po pierwsze obliczany jest pozostały RAM po AAC, po drugie ujemne RAM są ograniczane (curtailed) do zera, „low PTDF" poniżej określonego progu dla CNEC z niskimi RAM są ustawiane na zero, a dodatnie RAM są używane do obliczenia pozostałych dodatnich ATC dla rynku intraday. Na koniec wcześniej ograniczone ujemne RAM są używane do obliczenia ujemnego ATC.

Czynności (właściciel: CCC; czasy wg harmonogramu HLBP):

  • Rozdzielenie przepływów przed wyzwoleniem FBPCM dla ID ATC (Split flows before Trigger FBPCM for ID ATC).
  • Wyzwolenie FBPCM dla ID ATC (Trigger FBPCM for ID ATC) — narzędzie FBPCM.
  • Przetworzenie wyników ID ATC (Process ID ATC Results) — narzędzie FBPCM.
  • Dystrybucja wyników ID ATC do XBID (Distribute ID ATC Results to XBID) — NTCs; do XBID.
  • Dystrybucja wyników ID ATC (Distribute ID ATC Results) — ATCs, NTCs; do TSO.
  • Dystrybucja połączonej redukcji ATC (Distribute merged ATC Reduction) — wejścia: ATCs; wyjścia: Merged ATCs; do TSO.
  • Dystrybucja wyników końcowego FBC (Distribute Final FBC Results) — Final Domain; do TSO, JAO.

Do obliczenia ID ATC dla SIDC wymagane są następujące dane wejściowe:

  • Końcowe parametry flow-based dla każdego MTU procesu Core ID CC przy RefProg Balance, składające się z PTDF opartych na CGM oraz RAM dla wszystkich CNEC zgodnie z art. 21 ID CCM.
  • External Constraints, jeśli to konieczne.
  • AAC grid model i najnowszy AAC wysłany przez XBID.

Wyjścia to:

  • ID NTCs (ID3-921)
  • ID ATCs (ID3-922)
Spóźniona dostawa NTC (po 09:55) — Late NTC delivery (after 09:55)

Od wersji CCCT 4.2 możliwe jest przesłanie NTC do XBID po 09:55. W przypadku, gdy punkt kontrolny (checkpoint) dostarczenia NTC do XBID nie zostanie osiągnięty przed 09:53, proces IDCC(d) zostaje wstrzymany. Jest następnie wznawiany po zakończeniu IDA3. Wyzwalana jest nowa ekstrakcja ATC, która uwzględni AAC po IDA3. Nowe NTC zostaną dostarczone do XBID przed 10:30.

Eksport CGM przy już alokowanych zdolnościach (Export CGM at the Already Allocated Capacities)
