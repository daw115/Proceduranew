Obsługa procesu IDCC na punkcie — procedura skrócona

Szybka karta dla Stanowiska 5 i 6. Pełna procedura: Procedura_procesu_IDCC.html

Cykl obsługi

  1. Monitoruj pulpit CCM — kafelki każdego pliku (kolor = stan).
  2. Stanowisko 5: generacja/wysyłka (KreatorIDCF → Publikuj Connector; w razie backupu Manual Upload).
  3. Stanowisko 6: weryfikacja dostarczenia w CCM; braki/błędy → zgłoś.

Kolor → działanie

Kolor kafelkaDziałanie
ROK — brak działań (proces poprawny / status ręczny / po CET pozytywny).
Przedwczesne — monitoring, czekaj do TET.
Zbliża się TET — czujność: sprawdź źródło (ZP / Kreator / Perun4V / Core CC Tool).
!?Backup: odczytaj kod ACK / sprawdź Core CC Tool (Message Viewer); brak na MinIO → Manual Upload; brak na sFTP → wyślij ręcznie.
WW↑CETWysłany ręcznie z CCM — czekaj na ACK, ustaw 'R' po weryfikacji.
KRYTYCZNE — minął CET: zgłoś (CIZ/WPO/PSE-I) + telefon RCC; po CET brak lokalnego fallbacku.

Reguły kluczowe

Brak na MinIO → pobierz/wygeneruj ręcznie i Manual Upload (GUI Core CC Tool) / wyślij z CCM. Brak na sFTP/ZP → wyślij ręcznie ze źródła.

Backup GLSK/CBCORA/RA: KreatorIDCF → GLSK / CBCORA / RA → Publikuj Connector (procedura).

  • ACK 20 bramka zamknięta → admin Core CC Tool
  • ACK 21 wersja nie wyższa → podbij wersję i wyślij
  • ACK 30 rozmiar niezgodny → pobierz poprawny i wyślij
Zgłaszanie: CIZ, WPO, PSE-I. Stany krytyczne po CET → dodatkowo telefon RCC/TSCNET.
