Grupa I — Narzędzia dyspozytorskie (CCM)

R01 — Brak dostępu do CCM

Aplikacja: CCM (N01) | Źródło: HTML §9 R06 BUP DA §Ryzyko 27 (str. 69) | U-kod: U01
Wpływ na IDCC: brak monitoringu kafelków; brak ręcznej wysyłki przez PPM → Wyślij; brak funkcji „Informacje" o pliku; utrata wglądu w fazę procesu.

A. Procedura szczegółowa

  1. Diagnoza dostępności: sprawdź czy https://ccm.spsm.pse.pl ładuje się; spróbuj w innej przeglądarce / VPN.
    📷 CCM Instrukcja v2.6 — Okno logowania do aplikacji
  2. Zgłoś (CIZ, WPO, PSE-I).
  3. Tymczasowa praca bez CCM:
    • Core CC Tool Message Viewer → statusy plików
    • MinIO GUI (https://minio-gui.pse.pl) → bezpośredni podgląd bucketów
    • Perun4V → dla wyników FBA
    • kdm6@pse.pl → raporty CCA
  4. Ręczna wysyłka plików (gdy brak CCM): GUI CCCt Manual Upload bezpośrednio z dysku.
    📷 BUP DA — Core CC Tool Manual Upload (str. 47)
  5. Po przywróceniu CCM: wpis w karcie A.4.
CCM niedostępne → sprawdź VPN/przeglądarkę → zgłoś → tymczasowo: CCCt MV / MinIO / Perun4V / kdm6 → wysyłka przez P01 → P06

R02 — Błędna paczka do Perun4V (weryfikacja CCM)

Aplikacja: CCM (N01), Perun4V (N03) | Źródło: BUP DA §Ryzyko 28 (str. 69) | U-kod: U08, U09
Wpływ na IDCC: paczka ZIP (PERUN_IDx) niekompletna → niemożliwa konstrukcja paczki do Perun4V → brak obliczeń FBA.

A. Procedura szczegółowa

  1. Rozwiń agregat PERUN_IDx w CCM (kliknięcie ▼); zidentyfikuj problematyczną składową — kolor czerwony !, czarny lub czerwony.
  2. Sprawdzenie szczegółów: LPM na ikonę statusu → wybierz „Informacje" → odczytaj rozmiar i lokalizację MinIO.
    📷 CCM Instrukcja v2.6, Rys. 11 — Monitorowanie wielkości plików (s. 15)
  3. Pobranie poprawnego pliku z CCCt Message Viewer:
    • CCCt → MV → filtr po FID składowej (np. FID2-610 dla Merged GLSK)
    • ⚠ Uwaga: FIDx-610 (Merged GLSK z CCCt) ≠ FIDx-607 (raw GLSK od PSE)
    📷 BUP DA Rys. 11 — Core CC Tool, okno główne (s. 17)
    📷 BUP DA Rys. 12 — Message Viewer (s. 17)
  4. Wysyłka przez CCM: PPM na wierszu składowej → „Wyślij" → wskaż pobrany plik.
    📷 BUP DA Rys. 19 — CCM, rozwijane menu opcja Wyślij (s. 24)
  5. Po wysłaniu: składowa → zielony 'R'; agregat PERUN_IDx → zielony 'W'.
CCM rozwiń agregat → zidentyfikuj składową (czerwony!/czarny) → LPM Informacje → CCCt MV pobierz → CCM PPM Wyślij → zielony W

R04 — Brak dostarczonego pliku do CCCt

Aplikacja: Core CC Tool (N02) | Źródło: HTML §9 R04+R09 BUP DA §Ryzyko 12 (str. 48) | U-kod: U08, U11
Wpływ na IDCC: plik nie pojawił się w CCCt — proces nie kontynuuje, brak publikacji do PuTo/XBID.

A. Procedura szczegółowa

  1. Sprawdzenie w CCCt Message Viewer:
    • CCCt → selektor procesu obok niebieskiego pola → wybierz ID1 / ID2 / ID3C / ID3
    • Wybierz Business Day → Message Viewer
    • W polu Message Definition wpisz np. FID1-831 → Apply
    📷 BUP DA Rys. 12 — Message Viewer (s. 17)
    📷 BUP DA Rys. 13 — szczegóły pobieranego pliku (s. 18)
  2. Analiza wyniku:
    StatusDziałanie
    ProcessedCCM nie odświeżył; ustaw status ręcznie 'R' w CCM
    RejectedOdczytaj kod ACK → R05
    Brak wpisuPlik faktycznie nie dotarł → R18 (CN2)
  3. Pobranie ze źródła:
    • AC (FIDx-831): P02 — z ZP
    • IVA: P04 — z MinIO
    • Składowe: P05 — CCCt MV (inna iteracja jeśli możliwe)
  4. Manual Upload: CCCt → przycisk „Manual Upload" → wskaż plik → Upload file.
    📷 BUP DA Rys. 15 — Core CC Tool manual file upload (s. 20)
  5. Zweryfikuj status Processed → P06 wpis A.4.
CCCt MV → selektor procesu ID1...ID3 → Message Definition FID → Apply → brak wpisu = R15/R18 → P02/P04 → P01 → Processed → P06

R05 — Plik odrzucony przez CCCt (ACK negatywny)

Aplikacja: Core CC Tool (N02) | Źródło: HTML §9 R05 BUP DA §Ryzyko 13 (str. 49) | U-kod: U10, U12
Wpływ na IDCC: plik dotarł do CCCt, ale ACK negatywny — proces zatrzymany; konieczne ponowienie z poprawionym plikiem / wyższą wersją.

A. Procedura szczegółowa

  1. Odczyt kodu ACK w CCM: najedź na czerwony status w kolumnie „CCTool zwalid." → odczytaj komunikat.
    📷 CCM Instrukcja v2.6 — Komunikat ACK po najechaniu na status (s. 13)
  2. Weryfikacja w CCCt Message Viewer: CCCt → ID# process → MV → filtr po FID → wybierz wpis → szczegóły rejekcji.
  3. Reakcja per kod ACK:
    KodKomunikatDziałanie
    20Gate for provided message definition is not openedBramka zamknięta — sprawdź TET/CET; CCCt proces IDx-AC-FORWARDING; admin CCCt (zgłoś)
    21Provided version is not higher than already stored dataduplicatedVersion — podbij wersję; P02/P04 → P01 z v+1
    22Delivered constraint value is negative❌ Nieaktualne (CCCt akceptuje wartości ujemne)
    30File size mismatchP04 MinIO / P02 ZP → weryfikacja zawartości → P01
    40Backup version lowerNIE wysyłać backupu
    41Backup not available→ R23
    42Backup validation failedSprawdź raport CCA → zgłoś
  4. Po poprawie i ponownej wysyłce → zweryfikuj Processed → P06.
Najedź na czerwony status → odczytaj kod ACK → reakcja per kod (20: admin CCCt (zgłoś); 21: v+1 + P01; 30: P04/P02 → P01) → Processed → P06

R06 — Brak ACK / timeout

Aplikacja: Core CC Tool (N02) | Źródło: HTML §9 R05 (zawężenie) BUP DA §Ryzyko 12 (str. 48) | U-kod: U11
Wpływ na IDCC: plik wysłany do CCCt (CCTool wyśl. zielony), ale ACK nie napłynął w czasie (>2-3 min). Ryzyko duplikatu (kod 21).

A. Procedura szczegółowa

  1. Najedź na status czerwony ? → odczytaj czas oczekiwania.
    📷 CCM Instrukcja v2.6 — Brak ACK (czerwony ?), s. 12
  2. Czekaj 5 min — przeciążenie / opóźnienie transportu może być normalne.
  3. Po 5 min — CCCt Message Viewer:
    Status w CCCtDziałanie
    ProcessedCCM się nie odświeżył; ustaw status ręcznie 'R' w CCM; zgłoś
    Rejected→ R05
    Brak wpisu→ R18 (Connector 2.0)
  4. Brak wpisu >10 min: zgłoś; ponowna wysyłka z podbiciem wersji → P01 z v+1.
  5. ⚠ NIE wysyłać ponownie bez weryfikacji — kod 21 duplicatedVersion.
Najedź na '?' → odczytaj czas → czekaj 5 min → CCCt MV (Processed=R w CCM; Rejected=R05; brak=R18) → >10 min: zgłoś + P01 v+1

R07 — Niezakończenie walidacji ATC w czasie

Aplikacja: Core CC Tool — proces walidacji ATC | Źródło: HTML §9 R11, §5.4 | U-kod: U17 | Dotyczy: FIDx-928
Wpływ na IDCC: brak lokalnego fallbacku (HTML §5.4). Po CET dyspozytor nie wykonuje dalszych czynności.

A. Procedura szczegółowa

  1. PRZED CET — uzupełnienie składowych paczki ATC:
    • CCM → rozwiń agregat PERUN_IDx_ATC
    • Zidentyfikuj brakujące/błędne składowe (czarny/czerwony/czerwony !)
    • P05 — pobierz z CCCt MV → wyślij przez CCM
  2. Monitoring walidacji ATC: CCM kafelek FIDx-928; CCCt → ID# process → MV → filtr FID1-928.
  3. PO CET: ⛔ STOP — żadnych działań ręcznych.
    HTML §5.4 cyt.: „Brak wyniku ATC w czasie → dyspozytor nie wykonuje dalszych czynności. Brak lokalnego fallbacku."
  4. Zgłoszenie natychmiast: zgłoś + telefon CCC; P06.
Przed CET: uzupełnij składowe PERUN_IDx_ATC → P05 → CCM Wyślij → monitoruj FIDx-928. Po CET: ⛔ STOP — zgłoś + telefon CCC; P06

    KomunikatDziałanie

    Błędny plik w paczcePobierz wszystkie pliki z CCCt MV → przepakuj (R16)

R09 — ERR-I (część TS niepoliczona)

Aplikacja: Perun4V (N03) | Źródło: HTML §9 R02 BUP DA §Ryzyko 6 (str. 29-38) | U-kod: U15
Wpływ na IDCC: publikacja wyższej wersji IVA = wyniki Perun4V + dane CCA dla brakujących TS.

A. Procedura szczegółowa

  1. Identyfikacja w Perun4V GUI: status z dopiskiem ERR-I; kolumna Timestamps (Failed/Success/Running/Total).
    📷 BUP DA Rys. 31 — Perun4V ERR-I (s. 29)
    ⚠ Uwaga: 25 Timestamps = 24 godziny + 1 raporty
  2. Szczegóły per godzina: Status → Details → kolumna [Result].
    📷 BUP DA Rys. 32 — Perun4V niepoliczona godzina (s. 30)
  3. Weryfikacja w CCCt: czy zastosowano DFP (Default Flow Parameters) lub Spanning hours dla problematycznych godzin (Process Detail → klik w datę BD).
  4. Email: sprawdź kdm6@pse.pl — CCC wysyła informację o DFP/Spanning.
  5. Monitoring IVA BACKUP: kafelek powinien pokazać wyższą wersję (Backup > IVA z Perun4V). Stan red_pulse → Scenariusz 4 HTML §8.8.
  6. Brak publikacji Backup w czasie: zgłoś + telefon CCC.
  7. ⚠ Zakaz stosowania fallbacku 20% Fmax z DA.
Perun4V ERR-I → Status → Details → niepoliczone godziny → sprawdź DFP/Spanning w CCCt → kdm6 → monitoruj IVA Backup (wyższa wersja) → brak = zgłoś + telefon CCC

R10 — Process failed (wszystkie TS niepoliczone)
