Instrukcja dla bazodanowca — baza treści popupów IDCC

Import danych pulpitu CCM + renderowanie identyczne jak w Confluence (Storage Format).

1. Pliki źródłowe

PlikRekordyPoziomFormat
popup_content_IDCC.csv / .json126plik (1 wiersz pulpitu)CSV: UTF-8 BOM, separator ;, pola w " / JSON: tablica obiektów

Kodowanie wszystkich plików: UTF-8 (CSV z BOM dla Excela). Separator CSV: średnik ;. Każde pole ujęte w cudzysłów " (cudzysłowy wewnętrzne podwojone). Pola z HTML nie zawierają znaków nowej linii.

2. Schemat tabel (DDL — PostgreSQL)

CREATE TABLE fid_files (
  lp                INT PRIMARY KEY,
  kod               VARCHAR(40),
  obszar            VARCHAR(120),
  iteracja          VARCHAR(12),
  nazwa_przeplywu   VARCHAR(200),
  typ               VARCHAR(16),
  profil            VARCHAR(8),
  target_end_time   VARCHAR(5),      -- 'HH:MM'
  critical_end_time VARCHAR(5),
  wersja            INT,
  opis_pliku        TEXT,
  sciezka_pliku     TEXT,            -- zawiera HTML (linki)
  zrodlo            VARCHAR(120),
  kontakt           TEXT,
  popup_html        VARCHAR(120),
  uwagi             TEXT,
  confluence_storage TEXT            -- gotowy Storage Format
);

CREATE TABLE fid_states (
  wariant           VARCHAR(60),
  profil            VARCHAR(8),
  nr                INT,
  kategoria         CHAR(1),         -- A/B/C/D
  kategoria_nazwa   VARCHAR(24),
  kafelek1_kod      VARCHAR(16),
  kafelek1          VARCHAR(40),
  kafelek2_kod      VARCHAR(16),
  kafelek2          VARCHAR(40),
  opis_stanu        TEXT,
  procedura_html    TEXT,            -- HTML drzewka (linki)
  confluence_storage TEXT,
  PRIMARY KEY (wariant, nr)
);
MySQL: typy VARCHAR→jw., TEXT→MEDIUMTEXT, kolacja utf8mb4_unicode_ci. Oracle: VARCHAR2, CLOB dla pól HTML.

3. Import

PostgreSQL (CSV)

\copy fid_files FROM 'popup_content_IDCC.csv'
  WITH (FORMAT csv, DELIMITER ';', QUOTE '"', HEADER true, ENCODING 'UTF8');
\copy fid_states FROM 'popup_states_IDCC.csv'
  WITH (FORMAT csv, DELIMITER ';', QUOTE '"', HEADER true, ENCODING 'UTF8');

MySQL (CSV)

LOAD DATA LOCAL INFILE 'popup_content_IDCC.csv' INTO TABLE fid_files
  CHARACTER SET utf8mb4
  FIELDS TERMINATED BY ';' ENCLOSED BY '"'
  LINES TERMINATED BY '\n' IGNORE 1 LINES;
CSV ma BOM — przy MySQL/niektórych narzędziach usuń BOM lub użyj plików .json. W JSON pola confluence_storage, procedura_html, sciezka_pliku są gotowymi stringami HTML/XML.

4. Relacja plik ↔ stany

Stany są zdefiniowane per wariant (wspólne dla wielu plików). Plik łączy się ze stanami przez typ → wariant:

typ plikuwariant stanów (fid_states.wariant)liczba stanów
8312CCT — AC (źródło ZP)32
607, 6172CCT — GLSK/CBCORA (źródło Kreator)32
928928 — ATC Based Validation (zwrotny)18

Tabela mapująca (zalecane):

CREATE TABLE fid_typ_wariant (typ VARCHAR(16) PRIMARY KEY, wariant VARCHAR(60));
INSERT INTO fid_typ_wariant VALUES
 ('831','2CCT — AC (źródło ZP)'),
 ('607','2CCT — GLSK/CBCORA (źródło Kreator)'),
 ('617','2CCT — GLSK/CBCORA (źródło Kreator)'),
 ('928','928 — ATC Based Validation (zwrotny)');

-- popup pliku + jego macierz stanów:
SELECT f.kod, f.opis_pliku, s.nr, s.kafelek1, s.kafelek2, s.opis_stanu, s.procedura_html
FROM fid_files f
JOIN fid_typ_wariant m ON m.typ = f.typ
JOIN fid_states s    ON s.wariant = m.wariant
WHERE f.kod = 'FID3-607'
ORDER BY s.nr;
Macierze stanów istnieją dla profili z popupami: 2CCT (831/607/617) i 928. Profile 3COL (IVA 710), MIN / 2MS / SFTP nie mają jeszcze macierzy — te pliki mają na razie tylko warstwę plik-poziom (fid_files).

5. Pola z treścią HTML

  • opis_pliku, opis_stanu, zrodlo, kontakt — tekst (mogą zawierać znaki &, polskie znaki).
  • sciezka_pliku, procedura_html — HTML (linki <a href>, listy, <code>). Renderować jako HTML (nie escapować przy wyświetlaniu).
  • confluence_storage — Confluence Storage Format (XHTML + makra). Do bezpośredniego wstawienia na stronę Confluence.
Linki w polach HTML wskazują pliki lokalne (np. Inwentarz_IDCC.html#N02, Ryzyka_pelna_baza_IDCC.html#R05). Przed publikacją podmień bazowe adresy na URL-e docelowego systemu (strony Confluence / aplikacji).

6. Renderowanie identyczne jak w Confluence

Każdy rekord ma kolumnę confluence_storage w Confluence Storage Format — gotowa treść panelu, renderuje się identycznie jak w Confluence (makro panel + lozenże status dla kafelków).

Wstawienie na stronę Confluence (REST API)

PUT /rest/api/content/{pageId}
{
  "version": { "number": N+1 },
  "type": "page", "title": "...",
  "body": { "storage": { "value": "<wartość confluence_storage>", "representation": "storage" } }
}

Lub ręcznie: edytor strony → Insert > Markup > Confluence storage → wklej blok z popup_confluence_storage.xml.

Mapowanie kafelek → makro status (lozenża Confluence)

Kafelekcolourtitle
zielony / zielony R / ciemnozielony W / W po CETGreennazwa kafelka
czerwony / czerwony ! / czerwony ?Rednazwa kafelka
pomarańczowyYellowpomarańczowy
fioletowyBluefioletowy
szary / szary ? / czarny / pusteGreynazwa kafelka
Makro status w Confluence ma 5 kolorów (Grey/Red/Yellow/Green/Blue) — nie ma fioletu ani czerni. Odwzorowano najbliższym kolorem + tytuł = pełna nazwa kafelka (fioletowy→Blue, czarny→Grey). Jeśli front-end ma renderować dokładne kolory kafelków, użyj procedura_html/własnego komponentu zamiast makra status.
