1. Cel i zakres procedury

Niniejsza procedura określa sposób postępowania dyspozytora PSE w procesie Intraday Capacity Calculation (IDCC) dla regionu Core, obejmując:

  1. Przygotowanie danych wejściowych TSO (IGM, CB, GLSK, External Constraints)
  2. Monitoring przebiegu procesu w trybie Happy Day
  3. Użycie narzędzi wewnętrznych (CCM, MinIO, Perun4V, Core CC Tool)
  4. Dopuszczone działania backupowe w przypadku nieprawidłowości
  5. Postępowanie w sytuacjach ryzyka operacyjnego

Procedura dotyczy aktywnych iteracji IDCC: IDCC(a), IDCC(b), IDCC(c), IDCC(d) oraz widocznego w tabelach placeholdera IDCC(e) nieaktywne.

Zakres odpowiedzialności dyspozytora PSE Terminowe dostarczenie danych wejściowych · Monitoring walidacji FBA w Perun4V · Monitoring walidacji ATC · Interwencja backupowa w przypadku nieprawidłowości · Eskalacja do WPO (Słabicki, Jarzęcka, Krzysztoszek, Łukaszewski, Rodo)
🚫 Ograniczenia — PSE nie wykonuje PSE nie steruje fazami centralnymi procesu, nie wykonuje czynności należących do CCC oraz nie rozbudowuje ręcznie procesu ATC poza ewentualnym uzupełnieniem brakujących danych do paczki wejściowej.

⚡ Kluczowa różnica: IDCC(a) vs IDCC(b)–(d)

Iteracja IDCC(a) ma charakter odmienny — PSE nie dostarcza IGM, nie realizuje TSO Data Gathering, nie wykonuje walidacji FBA przez Perun4V. Zakres działania ogranicza się do monitorowania walidacji ATC (FID1-928).

3.2. Czynności wspólne przed rozpoczęciem monitorowania

  1. Otworzyć CCM pod adresem https://ccm.spsm.pse.pl/ i wybrać pulpit "IDCC FB Pulpit dyspozytorski".
  2. Potwierdzić poprawność monitorowanej doby biznesowej.
  3. Ustawić tryb odświeżania — w fazie krytycznej (przed TET/CET): szybki (30 s) lub ultraszybki (10 s).
  4. Zapewnić aktywny dostęp do Perun4V, MinIO, Core CC Tool oraz skrzynki kdm6@pse.pl.
  5. Zweryfikować, że poprzedni krok procesu centralnego został zakończony.

3.3. Zakres odpowiedzialności PSE w IDCC

  • Dostarczenie IGM do RCC (serwer Swissgrid) przed target run DACF/IDCF
  • Dostarczenie CB (FIDx-617) i GLSK (FIDx-607) do CCCt w fazie TSO Data Gathering
  • Weryfikacja poprawnego utworzenia paczek FBA i ATC w CCM
  • Nadzór nad automatycznym uruchomieniem obliczeń FBA w Perun4V
  • Śledzenie publikacji wyników FBA do CCCt i MinIO
  • Kontrola wyniku walidacji ATC (FIDx-928) w CCM
  • Weryfikacja automatycznej IVA BACKUP oraz ocena logiki publikacji
  • Publikacja AC (FIDx-831) do PuTo jako wynik procesu
  • Wykonanie działań backupowych wyłącznie w zakresie dopuszczonym procedurą

4.4. IDCC(a) — Happy Day

Faza 0: Przebieg centralny (bez udziału PSE)

Distribution of Initial Data → Calculate Default DA Domain → DACC/MCR Data Gathering → Calculate Enlarged Domain → Initial ID ATC Extraction. Na tym etapie CCA generuje FID1-928v1 z zerowymi wartościami (~10:00 D-1).

Faza 1: Walidacja ATC przez CCA (automatyczna)

  1. Trigger SDAC (po ~13:00 D-1): CCCt generuje domenę ATC "initial" z modelu D2CF i sald handlowych Core.
  2. Analiza CCA: Sprawdzenie przeciążeń na polskich CNECach (z uwzgl. transferów PL-SE, PL-LT).
  3. Publikacja FID1-928v2 (13:15–14:10 D-1): Wykryto przeciążenia → ograniczenie ATC. Brak przeciążeń → wartości z domeny initial.

Faza 2: Monitoring dyspozytora w CCM

  1. Odszukać wiersz ATC Based Validation (FID1-928). Po 13:15 D-1 — wersja v2, status zielony.
  2. Wersja v1 = fallback zerowy. Wersja v2+ = faktyczny wynik walidacji.
  3. Nie oczekiwać paczki do Perun4V, nie uruchamiać Perun4V.
  4. Brak v2 po 14:30 D-1 → sprawdzić MinIO → poinformować CCC → scenariusz R11.

Faza 3: Final ATC Extraction i zakończenie

  1. CCCt realizuje Final ID ATC Extraction z uwzgl. merged ATC (ID1-929).
  2. NTCs (ID1-921) dystrybuowane do XBID. Deadline: 15:00 D-1 (max 17:20 D-1).
  3. Dyspozytor nie wykonuje dalszych czynności. W razie błędu — poinformować WPO.
AAC Fallback Jeśli IDCC(a) zostanie zabity lub nie dostarczy computed AACs do CCCt, CCC informuje PSE o konieczności ręcznego dostarczenia AACs do XBID przez lokalne narzędzie (deadline: 20:00 D-1 dla IDCC(b)). Szczegóły: scenariusz R13.

5.2. IDCC(b)–(d) — Happy Day

Faza 1: Dostarczenie IGM do RCC

  1. PSE dostarcza IGM do TSCNET przed target run DACF/IDCF.
  2. Monitorować w CCM: IDx-613 (CGM), IDx-615 (Merging Response), IDx-751 (AAC Grid Model).

Faza 2: TSO Data Gathering

  1. PSE dostarcza do CCCt: FIDx-617 (CB), FIDx-607 (GLSK), FIDx-609 (EC opcjonalnie) przez ECP/EDX lub CCCt GUI.
  2. Sprawdzić w CCM pliki zwrotne: Merged CB, Merged GLSK, Merged EC, CGM DQC.
  3. Sprawdzić raport DQC — błędy dotyczące plików PSE → poprawić i wysłać ponownie → poinformować WPO.

Faza 3: Individual Validation (FBA)

  1. W CCM zweryfikować utworzenie agregatu paczki FBA i agregatu paczki ATC.
  2. Kolumna sFTP/ZP wysł. dla agregatu FBA — zielona = paczka przekazana do Perun4V.
  3. W Perun4V potwierdzić pojawienie się obliczenia uruchomionego przez sFTP@pse.pl.
  4. Po zakończeniu obliczeń: potwierdzić publikację wiersza IVA FBA (FIDx-710), raportów i dostarczenia do CCCt/MinIO.
  5. Niezależnie monitorować wiersz FIDx-928 — wynik walidacji ATC po stronie CCA.
  6. Wiersz IVA BACKUP — monitoring wyłącznie w zakresie potwierdzenia automatycznej logiki.

5.4. Zasady szczególne dla ATC

  • Walidacja ATC jest strumieniem odrębnym od FBA.
  • Happy Day: monitoring poprawności paczki ATC → monitoring publikacji FIDx-928.
  • Backup: dopuszczalne wyłącznie uzupełnienie brakujących danych do paczki ATC w CCM.
  • Brak wyniku ATC w czasie → dyspozytor nie wykonuje dalszych czynności. Brak lokalnego fallbacku.

6.5. CB file — Critical Branches (FIDx-617)

Plik CB (CBCORA) zawiera listę CNEC-ów (Critical Network Element and Contingency) zdefiniowanych przez PSE. Każdy CNEC = CNE + Contingency + Remedial Actions.

Format pliku CB W standardowym przebiegu plik CB jest generowany automatycznie przez kreatory PSE w formacie XML (format natywny CCCt) i wysyłany przez ECP/EDX.

Format XLSX jest formatem awaryjnym/historycznym — dyspozytor może się z nim zetknąć przy awarii kreatora, korekcie pliku CB przez specjalistę spoza zespołu eksploatacji albo otrzymaniu pliku CB od strony trzeciej. W takich przypadkach wymagana jest konwersja XLSX → XML przez CCCt CBCORA Converter (dostępny w CCCt GUI lub REST API).

8.8. Scenariusze CCM

SCENARIUSZ 1
Monitoring Happy Day — IDCC(x)
Stan normalny. Wszystkie składowe zielone, paczka wysłana do Perun4V, obliczenia zakończone, FIDx-710 przyjęte przez CCCt.
  1. Otworzyć pulpit "IDCC FB Pulpit dyspozytorski" · ustawić dobę
  2. Rozwinąć agregat PERUN_IDx → wszystkie składowe zielone
  3. Kolumna sFTP/ZP wysł. dla PERUN_IDx — zielona
  4. W Perun4V: obliczenie uruchomione przez sFTP@pse.pl
  5. Po zakończeniu: CCM wiersz FIDx-710 → "CCCTool wysł." zielony
  6. CCCTool zwalid. — zielony (Processed)
  7. Sprawdzić wiersz FIDx-928 (ATC)
  8. Sprawdzić wiersz FIDx-831 (AC) — zweryfikować publikację wyniku do PuTo i status Processed w Core CC Tool
  9. Sprawdzić wiersz IVA BACKUP — zweryfikować wersję i status (zielony z pulsującą obwódką)
SCENARIUSZ 2
Brakująca składowa paczki FBA
Czerwony ! na składowej (np. FIDx-610 — Merged GLSK, scalony plik GLSK z CCCt — za mały). Agregat PERUN_IDx nie może zostać wysłany.
UwagaFIDx-610 (Merged GLSK) różni się od FIDx-607 (raw GLSK dostarczanego przez PSE w fazie TSO Data Gathering). Merged GLSK jest plikiem zwrotnym z CCCt zawierającym scalone GLSK wszystkich TSO.
  1. Rozwinąć agregat → zidentyfikować problematyczną składową
  2. Kliknąć kafelek → "Informacje" → odczytać szczegóły błędu
  3. Pobrać prawidłowy plik z CCCt Message Viewer lub MinIO
  4. Wysłać ręcznie przez CCM (PPM → Wyślij)
  5. Zweryfikować zmianę na zielony-R; agregat → zielony-W
  6. Kolumna sFTP/ZP wysł. → zielona (paczka wysłana do Perun4V)
  7. W razie problemów → poinformować WPO
SCENARIUSZ 3
Ręczna wysyłka FIDx-710
Perun4V zakończył obliczenia (wynik w MinIO — zielony), ale CCM nie wysłał automatycznie do CCCt (szare kafelki).
  1. Zweryfikować w Perun4V, że obliczenia FBA zakończone (status "Successful" = wszystkie TS policzone, "ERR-I" = część TS niepoliczona, lub "Process failed" = żadne TS niepoliczone)
  2. Pobrać FIDx-710 z MinIO: perun/ID_FBCC/out/yyyy-mm-dd/
  3. Zapisać w katalogu sieciowym \YYYYMMDD\Perun\
  4. CCM → wiersz FIDx-710 → PPM → "Wyślij" → załadować plik
  5. Kolumna "CCCTool wysł." → zielony-W
  6. Kolumna "CCCTool zwalid." → zielony (Processed)
  7. Brak ACK po 5 min → manual upload do CCCt
SCENARIUSZ 4
Monitoring IVA BACKUP
Nowa wersja IVA BACKUP (~5 min przed CET). Zielony kafelek z pulsującą czerwoną obwódką. Wersja IVA BACKUP wyższa niż FIDx-710.
  1. ~5 min przed CET: sprawdzić wiersz IVA BACKUP
  2. Pulsujący zielony → odczytać numer wersji
  3. W Perun4V: sprawdzić wersję FIDx-710
  4. Porównać: IVA BACKUP > FIDx-710 = prawidłowe
  5. MinIO cca/ID_FBCC/out/BD/ → raport CCA dostępny
  6. Wersje niezgodne → poinformować WPO + eskalacja do CCC
  7. NIE uruchamiać ręcznej wysyłki IVA BACKUP
SCENARIUSZ 6
Brak ACK — kafelek czerwony ?
Plik wysłany do CCCt (zielony w "CCTool wysł."), ale brak ACK po 5 minutach.
  1. Najechać kursorem na "?" → odczytać czas oczekiwania
  2. CCCt → Message Viewer → filtr po kodzie przepływu
  3. Status "Processed" → CCM się nie odświeżył; czekać lub odświeżyć ręcznie
  4. Status "Rejected" → odczytać kod błędu:
    duplicatedVersion → podbić wersję i ponowić
    gateClosed → sprawdzić CET
    Negative constraint value → poinformować WPO natychmiast
  5. Brak wpisu po 10 min → poinformować WPO + zgłosić problem Connector 2.0
Uwaga Czerwony "?" ≠ brak pliku. Nie wysyłaj pliku ponownie — podwójna wysyłka = błąd duplicatedVersion.
SCENARIUSZ 13 — KRYTYCZNY
Agregat czarny — żadna składowa nie dotarła przed CET
Wszystkie składowe czarne. Paczka ZIP nie wygenerowana. Obliczenia FBA nie uruchomione. Wynik FIDx-710 niedostępny.
  1. Zidentyfikować wszystkie czarne wiersze agregatów PERUN_IDx
  2. Czy problem dotyczy jednej iteracji czy wszystkich? (awaria systemowa?)
  3. Sprawdzić Connector 2.0 — procesy dla danej doby
  4. Sprawdzić MinIO — czy pliki składowych w ogóle dotarły
  5. Skontaktować się z nadawcą każdej brakującej składowej
  6. Składowe dostępne → "wyślij po CET" → paczka ZIP odtworzona (status fioletowy)
  7. Składowe niedostępne → poinformować WPO natychmiast + eskalacja do CCC
  8. Po dostarczeniu: sprawdzić uruchomienie obliczeń w Perun4V
🚫 Scenariusz krytyczny Priorytetem jest kontakt z CCC przed podjęciem jakichkolwiek działań technicznych.

9. Ryzyka i postępowanie

R01
Brak automatycznego uruchomienia FBA po wysłaniu paczki
Objaw: Paczka wysłana (sFTP/ZP zielona), ale brak obliczenia w Perun4V
Działanie: Pobrać ZIP → zapisać → wgrać ręcznie do Perun4V → uruchomić → ręcznie wysłać FIDx-710
R02
ERR-I — część TS niepoliczona
Objaw: Status ERR-I w Perun4V
Działanie: Ustalić zakres niepoliczonych TS → monitorować IVA BACKUP w wyższej wersji → brak → WPO + CCC
R03
Process failed — brak wyników dla wszystkich TS
Objaw: Status Process failed w Perun4V
Działanie: Oczekiwać IVA BACKUP v1. Brak → WPO + CCC. Nie stosować fallbacku 20% Fmax.
R04
Brak paczki lub błędna składowa
Objaw: Agregat czarny / czerwony / czerwony !
Działanie: Rozwinąć agregat → "Informacje" → pobrać właściwy plik z CCCt/MinIO → wysłać przez CCM
R05
Brak ACK lub odrzucenie FBA
Objaw: Brak ACK lub Rejected w Message Viewer
Działanie: Sprawdzić nazwę/datę/wersję. Manual Upload do Core CC Tool dla wyższej wersji.
R06
Brak dostępu do CCM
Działanie: MinIO do pobierania plików + Core CC Tool do weryfikacji. Manual Upload do CCCt dla FBA.
R07
Brak dostępu do Core CC Tool
Działanie: Monitorować w CCM; telefonicznie potwierdzać z CCC. Alternatywny kanał wyłącznie po uzgodnieniu z CCC.
R09
Konieczność ręcznego uploadu FBA
Działanie: Manual Upload w Core CC Tool dla FIDx-710 → sprawdzić status Processed w Message Viewer.
R11
Niezakończenie walidacji ATC w czasie
Objaw: Brak FIDx-928 do końca czasu procesu
Działanie: Uzupełnić brakujące składowe tylko PRZED utratą sensu procesowego. Po przekroczeniu — brak działań ręcznych. Poinformować WPO.
R12
Niezgodność logiki IVA BACKUP z Perun4V
Działanie: Porównać wersje → niezwłocznie WPO + CCC + CCA. Nie zastępować automatyki ręcznym menu.
R13
AAC Fallback — IDCC(a) nie dostarczył AACs
Działanie: Dostarczyć computed AACs do XBID przez narzędzie lokalne. Deadline: 20:00 D-1 (dla IDCC(b)), 02:00 D (dla IDCC(c)) lub 07:00 D (dla IDCC(d)). Potwierdzenie mailowe do CCC.
R14
Decoupling SDAC — konieczność MD250
Działanie: Dostarczyć Shadow Auction Results (MD250) do CCCt przez ECP/EDX. Deadline 1: TET MCR (~15:43). Deadline 2: ~17:24.
R15
Late NTC delivery po IDA2 — IDCC(b) opóźniony
Objaw: Checkpoint dostarczenia NTC do XBID przed startem aukcji IDA2 nie został osiągnięty przed 21:53 D-1; proces IDCC(b) jest wstrzymany automatycznie.
Działanie: Automatyczne wznowienie po IDA2. Monitorować wznowienie i publikację NTC przed 22:30.
R16
Connector 2.0 nie kopiuje IVA do bucketów CCA
Działanie: Pliki w perun/…, ale nie w cca/… → WPO + PSE Innowacje. Brak w obu → WPO + CCA.
R20
Brak dostępnego raportu CCA
Objaw: Raport CCA nie został wygenerowany lub nie jest dostępny w bucket MinIO cca/ID_FBCC/out/BD/.
Działanie: Sprawdzić status procesu CCA w CCM. Jeśli raport nie jest dostępny mimo poprawnego zakończenia procesu — poinformować WPO i eskalować do CCA.
